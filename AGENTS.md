# AGENTS.md - pulumi-formal

Guide for humans and agents working on this repository.

## What This Repo Is

This is a **Pulumi provider** for [Formal](https://joinformal.com), bridged from the upstream [Terraform provider](https://github.com/formalco/terraform-provider-formal) using [pulumi-terraform-bridge](https://github.com/pulumi/pulumi-terraform-bridge). It generates SDKs for Python, Node.js, Go, and .NET.

The bridged architecture means we don't write resource implementations directly. Instead, we wrap the upstream Terraform provider and auto-generate Pulumi SDKs from its schema.

## Repository Layout

```
.ci-mgmt.yaml          # Config for Pulumi's CI management tool (regenerates workflows)
.config/mise.toml       # Tool versions (Go, Node, Python, etc.)
Makefile                # Build automation (autogenerated by ci-mgmt)
provider/
  resources.go          # Bridge configuration: maps TF resources to Pulumi
  cmd/
    pulumi-resource-formal/  # Provider binary entry point
    pulumi-tfgen-formal/     # Code generation entry point
upstream/               # Git submodule -> formalco/terraform-provider-formal
sdk/                    # Generated SDKs (dotnet, go, nodejs, python)
examples/               # Example Pulumi programs
.github/
  workflows/            # CI workflows (mostly autogenerated by ci-mgmt)
  actions/esc-action/   # Stub action for secret passthrough (ignore)
```

## Workflow

**All changes should be made in pull requests, not pushed directly to `main`.** The primary CI workflow (`run-acceptance-tests.yml`) triggers on PRs. Push-triggered workflows (`main.yml`, `prerelease.yml`, `release.yml`) only run on `main` and will show spurious failures on feature branches.

## Common Tasks

### Upgrading the upstream provider or the bridge

This is the most frequent maintenance task — either the upstream Formal TF provider
releases a new version, or `pulumi-terraform-bridge` needs a bump (e.g. the current
version doesn't support a new upstream feature like write-only attributes).

**Use Pulumi's [`upgrade-provider`](https://github.com/pulumi/upgrade-provider) CLI** —
the same tool the Pulumi bots run automatically on their own provider repos. It updates
the submodule and/or bridge, keeps the `replace` directive for the Pulumi fork of
`terraform-plugin-sdk/v2` in sync, aligns `pulumi/pkg`/`pulumi/sdk`, runs `go mod tidy`,
then `make tfgen` + `make generate_sdks`, and commits the result.

`upgrade-provider` is pinned in the root `mise.toml`, so `mise install` puts it on PATH.

```sh
# Upgrade the upstream TF provider (infers latest release if no --target-version):
upgrade-provider formalco/pulumi-formal --kind=provider --repo-path=.

# Upgrade the bridge only (optionally pin a version or git ref):
upgrade-provider formalco/pulumi-formal --kind=bridge --repo-path=. --target-bridge-version=v3.131.0

# Both at once:
upgrade-provider formalco/pulumi-formal --kind=all --repo-path=.
```

`--repo-path=.` makes it operate on the current checkout rather than cloning a fresh copy
into `$GOPATH`. It commits the changes directly onto your current branch (using your local
git identity, with messages `make tfgen` and `make generate_sdks`).

**Expected "failure":** the very last step tries to open a GitHub PR via Pulumi-internal
bot credentials we don't have, so it exits non-zero there. That's fine — the upgrade work
is already committed locally by that point. Verify with `git log` and a quick `make provider`
build, then open the PR yourself.

> Why this beats bumping by hand: changing the bridge version alone leaves the
> `terraform-plugin-sdk/v2` fork `replace` directive stale, which surfaces as confusing
> build/tfgen errors like `*GRPCProviderServer does not implement tfprotov5.ProviderServer
> (missing method GenerateResourceConfig)` or `pcl.Resource has no field or method
> DecomposeToken`. `upgrade-provider` keeps these aligned for you.
>
> The repo also ships `upgrade-bridge.yml` / `upgrade-provider.yml` GitHub workflows
> (autogenerated by ci-mgmt), but they depend on Pulumi's ESC secrets and provider-automation
> GitHub App, which aren't available to this repo. Running the CLI locally is the equivalent
> that actually works for us.

#### Doing it manually

If `upgrade-provider` is broken or unavailable, do the upstream-provider bump by hand:

1. **Update the submodule:**
   ```sh
   cd upstream
   git fetch
   git checkout <new-tag-or-commit>
   cd ..
   ```

2. **Update Go dependencies** (the provider imports the TF provider as a Go module):
   ```sh
   cd provider
   go get github.com/formalco/terraform-provider-formal@<version>
   go mod tidy
   cd ..
   ```
   If you're also bumping the bridge, bump `github.com/pulumi/pulumi-terraform-bridge/v3`
   here too, and update the `replace github.com/hashicorp/terraform-plugin-sdk/v2 =>
   github.com/pulumi/terraform-plugin-sdk/v2 <pseudo-version>` directive to match the fork
   the bridge's own `go.mod` pins (check `$GOMODCACHE/.../pulumi-terraform-bridge/v3@<ver>/go.mod`).

3. **Regenerate the schema and SDKs:**
   ```sh
   make tfgen          # Runs pulumi-tfgen-formal to generate schema
   make provider       # Builds the provider binary
   make build_sdks     # Builds all language SDKs
   ```

4. **Commit everything** - the generated SDK files, updated `go.mod`/`go.sum`, schema JSON, and the submodule pointer.

### Updating ci-mgmt (CI workflows)

The CI workflows are **autogenerated** by [pulumi/ci-mgmt](https://github.com/pulumi/ci-mgmt). To regenerate them after editing `.ci-mgmt.yaml`:

```sh
make ci-mgmt
```

**Warning:** This overwrites all workflow files, the Makefile, and other CI infrastructure. Any manual edits to these files will be lost. If you need to customize workflows, you may need to edit them after running `make ci-mgmt` and accept that the next regeneration will overwrite your changes.

### Building locally

```sh
make build              # Build everything (provider + all SDKs)
make provider           # Build just the provider binary
make schema             # Generate just the schema
make build_sdks         # Build all SDKs
make test               # Run example tests (must build first)
make test_provider      # Run provider unit tests
make lint_provider      # Run golangci-lint
```

Tool installation is handled by [mise](https://mise.jdx.dev/). The `Makefile` will run `mise install` automatically.

There are two mise config files: `.config/mise.toml` is **autogenerated by ci-mgmt** (don't edit — `make ci-mgmt` overwrites it), while the root `mise.toml` is the **repo-owned override** where local tool additions go. For example, the `upgrade-provider` CLI (see [Upgrading the upstream provider or the bridge](#upgrading-the-upstream-provider-or-the-bridge)) and the `golangci-lint` pin live in root `mise.toml`.

### Cutting a release

To release a new version, tag `main` and push the tag:

```sh
git tag v1.2.0
git push origin v1.2.0
```

That's it. The `release.yml` workflow triggers on semver tags (`v*.*.*`), builds the provider from scratch, runs all tests, and if they pass, automatically:
- Creates a GitHub Release with provider binaries for all platforms
- Publishes Python SDK to PyPI (`pulumi_formal`)
- Publishes Node.js SDK to npm (`@formalco/pulumi`)
- Publishes .NET SDK to NuGet
- Publishes Go SDK via a tagged commit on the `sdk` branch

For prereleases, use a semver prerelease tag (e.g., `v1.2.0-alpha.1`), which triggers `prerelease.yml` instead.

Java SDK publishing is disabled (see `.ci-mgmt.yaml`).

Note: the Pulumi provider version is intentionally kept separate from the upstream
`terraform-provider-formal` version, and the two will not match. Pulumi SemVer describes the
generated SDKs (a breaking SDK change bumps the Pulumi major regardless of upstream), and the Go
module path is pinned at `major-version: 1` and decoupled from upstream's major (see
`.ci-mgmt.yaml`). To keep the mapping discoverable, each GitHub release body records which upstream
TF version it wraps (see the `Get upstream provider version` step in `publish.yml`, under Known
issues).

## CI Architecture

### Workflow structure

Caller workflows (`main.yml`, `prerelease.yml`, `release.yml`, `run-acceptance-tests.yml`) invoke reusable workflows (`prerequisites.yml`, `lint.yml`, `license.yml`, `build_provider.yml`, `build_sdk.yml`, `test.yml`, `publish.yml`).

### Known issues and workarounds

- **ci-mgmt breaking changes:** The upstream `pulumi/ci-mgmt` tool introduces changes that assume Pulumi-internal infrastructure (e.g., a GitHub App for `create-github-app-token`). We've removed this dependency and use `secrets.GITHUB_TOKEN` instead. Running `make ci-mgmt` will re-introduce it, so you'll need to re-apply the removal afterward.

- **`golangci-lint` version:** Pinned in the root `mise.toml` (overriding the autogenerated `.config/mise.toml`). When Go is upgraded, the lint version may need to be updated to match. See commit `5f28d17` for an example of this.

- **Upstream version in release notes:** `publish.yml` has a repo-owned `Get upstream provider version` step that reads the `terraform-provider-formal` pin from `provider/go.mod` and prepends a "Wraps upstream terraform-provider-formal vX.Y.Z" line to the GitHub release body. Running `make ci-mgmt` overwrites `publish.yml`, so re-apply this step (and the `body:` reference to its output) afterward — same class of manual patch as the `GITHUB_TOKEN` one above.

## Provider Configuration

Key settings in `provider/resources.go`:
- Upstream provider: `github.com/formalco/terraform-provider-formal/formal`
- Token mapping: `tokens.SingleModule("formal_", "index", ...)` (all resources in one module)
- Plugin download: `github://api.github.com/formalco` (GitHub Releases)
- npm package: `@formalco/pulumi`
- PyPI package: `pulumi_formal`
- NuGet namespace: `Formal.Pulumi`
